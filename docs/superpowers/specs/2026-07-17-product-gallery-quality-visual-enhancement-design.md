# 商品画廊数据质量可视化增强设计

## 背景

当前 `web-scraping.dev` 商品采集链路已经完成真实端到端验收，静态画廊能够展示商品卡片、价格、状态和证据侧栏。现有真实样例包含 2 条 `PARTIAL` 记录，原因是详情页没有提供可靠的商品分类；截图中虽然能看到“部分”统计，但用户不容易理解部分完整的具体原因，生成时间也存在换行和阅读不顺的问题。

本轮目标是增强“数据质量优先”的可视化表达，让用户能够快速回答：

1. 当前采集结果有多少条完整、部分完整或失败记录？
2. `PARTIAL` 的原因是什么、缺少哪些字段？
3. 哪些商品属于“未分类”？
4. 当前卡片与证据侧栏是否来自同一份离线快照？

## 目标

- 在页面顶部增加数据质量摘要，明确展示总数、成功、部分、失败和缺失字段概况。
- 将 `PARTIAL` 状态与具体缺失字段关联显示，不让状态徽标成为孤立颜色。
- 在分类筛选中增加“未分类”选项，正确覆盖空 `category` 值。
- 优化生成时间、状态徽标、卡片密度和证据侧栏的信息层级。
- 保持现有深色视觉基调、静态离线打开能力和本地数据内嵌方式。

## 非目标与不变量

- 不修改商品采集器、浏览器访问规则或 live 授权门禁。
- 不修改 `products.xlsx` 的列契约、`products.json` 的 schema 或商品领域模型。
- 不为缺失分类推断或伪造分类值；空分类必须显示为“未分类”。
- 不新增 CDN、远程图片、字体、脚本、XHR、`fetch` 或其他打开页面时的网络请求。
- 不把运行时输出、截图、profile 或诊断 HTML 纳入 Git。

## 方案选择

### 方案 A：轻量摘要

只调整时间格式、卡片尺寸和状态颜色。实现成本最低，但不能解释 `PARTIAL` 的原因，无法解决当前截图中最核心的数据质量理解问题。

### 方案 B：数据质量优先（采用）

围绕现有 `ProductStatus`、`error_message` 和空字段构建质量摘要、缺失字段提示、“未分类”筛选及证据侧栏增强。它不需要改变数据 schema，却能直接改善当前真实采集结果的可读性和可操作性。

### 方案 C：分析面板优先

增加价格分布、分类分布和状态图表。它适合更大数据量，但在当前只有 2 条记录时信息密度不足，且会把重点从“理解数据质量”转移到统计展示。

## 交互与视觉设计

### 1. 顶部数据质量摘要

保留现有总计、成功、部分、失败和生成时间卡片，并增加一个“数据质量”卡片或摘要行：

- 完整度：`success / total`；
- 部分完整：`partial`；
- 失败：`failed`；
- 缺失字段：按字段聚合，例如“缺失分类 2”。

当没有缺失字段时显示“字段完整”；有缺失时使用琥珀色提示，但不使用错误红色替代真实失败状态。

### 2. `PARTIAL` 状态表达

商品卡片继续显示状态徽标，同时在徽标旁或卡片底部显示简短原因，例如：

```text
部分 · 缺少分类
```

失败记录显示 `error_message` 的摘要；完整记录不显示多余警告。状态颜色必须同时配合文字，不能只依赖颜色传达含义。

### 3. 分类筛选

分类下拉框加入固定选项“未分类”。筛选逻辑约定：

- 非空 `category` 按原值分组；
- 空字符串、空白字符串和缺失值统一归入“未分类”；
- “所有类别”仍显示全部记录；
- 筛选后顶部统计应同步反映当前可见记录数量，但保留原始总量作为上下文。

### 4. 证据侧栏

侧栏保留现有名称、描述、价格、品牌、变体数、产品 ID、来源 URL、状态和采集时间，并新增：

- “数据质量”小节；
- 缺失字段列表；
- 对 `PARTIAL` 的原始原因说明；
- “未分类”明确显示为标签，而不是孤立的破折号。

来源 URL、产品 ID 和采集时间继续使用等宽或高对比样式，方便复制与核对。

### 5. 布局与响应式行为

- 生成时间使用单行本地化格式，例如 `2026-07-17 12:42:44 (+08:00)`；窄屏时允许自然换行，但不得按字符拆分。
- 商品卡片网格使用可伸缩列数；只有少量商品时保持卡片可读宽度，避免内容挤在左侧后产生过大的空洞感。
- 证据侧栏在宽屏固定宽度，在窄屏移动到商品列表下方。
- 保持当前深色背景和高对比状态色，不引入新的视觉主题。

## 技术设计

### 数据流

```text
products.json
  -> render_gallery()
  -> 页面初始化时计算质量摘要、缺失字段和分类选项
  -> 搜索/筛选/排序只作用于内嵌快照
  -> 点击卡片更新证据侧栏
```

所有新增统计都从已内嵌的 `ProductCollection` 数据派生，不建立第二份数据源，也不在浏览器打开后访问目标网站。

### 代码边界

- 主要修改：`app/product_gallery.py`；
- 主要测试：`tests/test_product_gallery.py`；
- 如需静态契约测试，扩展 `tests/test_project_config.py`；
- 不修改 `app/product_models.py`、`app/product_json.py` 或采集器。

### 可访问性

- 状态徽标包含可读文本和适当的 `aria-label`；
- 筛选控件使用原生 `label` 或可访问名称；
- 键盘焦点在卡片和侧栏控件上清晰可见；
- 失败/部分原因使用文本呈现，不依赖颜色或图标单独表达。

## 验收标准

### 自动化验收

- 现有全量测试继续通过；
- gallery 测试验证：
  - 顶部数据质量摘要包含正确数量；
  - `PARTIAL` 记录显示缺失字段和原始原因；
  - 空分类生成“未分类”选项并可筛选；
  - 生成时间不会按字符拆分；
  - HTML 不包含外部脚本、字体、`fetch`、XHR 或 WebSocket；
  - 所有统计来自同一份内嵌 JSON 数据。

### 人工验收

使用当前 2 条 `PARTIAL` 真实产物打开 `gallery.html`，确认：

- 顶部能一眼看出“总计 2、部分 2、失败 0”；
- 页面明确提示两条记录都缺少分类；
- 选择“未分类”后只显示两条商品；
- 点击任一卡片，证据侧栏显示缺失字段和来源 URL；
- 生成时间完整可读；
- 断网或直接双击 HTML 时交互仍可用。

## 交付与回滚

- 交付物仅包括 gallery 渲染代码、自动化测试和本 spec/实施计划文档；
- 不提交 `outputs/`、`artifacts/` 或 `browser-profile/` 中的运行时文件；
- 如果视觉增强导致现有静态契约测试失败，优先回滚展示层改动，不回滚已验证的采集器修复。

## 后续实施顺序

1. 先为质量摘要、缺失字段和“未分类”筛选补失败测试；
2. 实现最小 HTML/CSS/JavaScript 变化；
3. 运行全量离线测试和 bundle verifier；
4. 用当前真实产物做本地人工验收；
5. 如有必要，再单独规划分析图表，不与本轮质量表达混合。
